## РОЛИ ПОЛЬЗОВАТЕЛЕЙ

```
SUPER_ADMIN → ADMIN → TEAM_LEAD → EMPLOYEE → INTERN
```

| Роль | Описание |
|------|---------|
| SUPER_ADMIN | Полный доступ ко всему, управляет ADMIN-ами |
| ADMIN | Управляет сотрудниками, дежурствами, новостями, отчётами |
| TEAM_LEAD | Видит свою группу + свои отчёты; функционал как у EMPLOYEE в приложении |
| EMPLOYEE | Стандартный сотрудник |
| INTERN | Стажёр — те же права в приложении, что у EMPLOYEE |

**Важно:** TEAM_LEAD, EMPLOYEE, INTERN — одинаковый доступ в мобильном приложении. Различие только в отображении роли и в том, что тимлид видит отчёты своей группы.

## СТАТУСЫ ПОЛЬЗОВАТЕЛЯ

| Статус | Описание |
|--------|---------|
| PENDING | Зарегистрировался, ждёт подтверждения админа |
| ACTIVE | Активный сотрудник |
| LEAVE | В отпуске (заявка одобрена) |
| WARNING | Получил предупреждение (опоздания / ранний уход) |
| BLOCKED | Заблокирован админом |
| DELETED | Мягкое удаление (данные сохраняются в БД для истории) |

## СТАТУСЫ ПОСЕЩАЕМОСТИ

**Приход (check_in_status):** `ON_TIME` | `LATE` | `EARLY_ARRIVAL`

**Уход (check_out_status):** `ON_TIME` | `LEFT_EARLY` | `OVERTIME`

**Общий статус дня (status):** `PRESENT` | `LATE` | `ABSENT` | `INCOMPLETE` | `APPROVED_ABSENCE` | `MANUAL` | `EARLY_LEAVE` | `OVERTIME`

## МОДУЛЬ 1 — АУТЕНТИФИКАЦИЯ И ПОЛЬЗОВАТЕЛИ

### 1.1 Регистрация сотрудника
- Сотрудник сам регистрируется через мобильное приложение
- Поля: ФИО, email (Gmail), телефон, логин, пароль
- После регистрации статус → `PENDING`
- Админу приходит push-уведомление: "Новый сотрудник ожидает подтверждения: [ФИО]"
- Админ подтверждает → статус → `ACTIVE`
- Админ отклоняет → пользователь получает уведомление об отклонении

### 1.2 Безопасность
- Пароль: bcrypt, cost factor **12**
- Access Token: **15 минут** (JWT)
- Refresh Token: **30 дней** (хранится в Redis)
- UUID как primary key везде

### 1.3 Управление пользователями (Admin)
- Просмотр списка всех сотрудников с фильтрами (роль, статус, группа)
- Изменение роли и статуса
- Просмотр профиля сотрудника (история посещаемости, дежурств)
- Мягкое удаление (soft delete через `deleted_at`) — данные сохраняются для истории
- Добавление Admin-комментария к профилю
- Назначение группы / тимлида (mentor_id)

### 1.4 Заявки на отсутствие (Leave Requests)
- Сотрудник подаёт заявку через приложение: даты + причина
- Админ одобряет → статус сотрудника → `LEAVE`
- Админ отклоняет → статус остаётся `ACTIVE`
- При одобрении дни автоматически помечаются как `APPROVED_ABSENCE` в посещаемости

## МОДУЛЬ 2 — ПОСЕЩАЕМОСТЬ

### 2.1 Принцип верификации (двойная проверка)
Приход/уход засчитывается ТОЛЬКО при одновременном выполнении двух условий:
1. **QR-код** — совпадает с активным QR офиса
2. **IP-адрес** — устройство в офисной сети (из таблицы `office_networks`)

### 2.2 Индивидуальный график (EmployeeSchedule)
- Для каждого сотрудника создаётся расписание по дням недели (Пн–Вс)
- Каждый день: `is_working_day` + `start_time` + `end_time`
- Минимальный рабочий день: **6 часов** (система не позволяет сохранить расписание с продолжительностью менее 6 ч)
- Выходной день: `is_working_day = false`, времена не указываются
- График настраивает **Admin** для каждого сотрудника

### 2.3 Приход (Check-In)
Сотрудник сканирует QR через мобильное приложение:
- Система проверяет QR + IP
- Система берёт расписание сотрудника на сегодня
- Определяет статус прихода:
  - Пришёл раньше → `EARLY_ARRIVAL` + "пришёл на X мин/ч раньше"
  - Пришёл вовремя (±5 мин) → `ON_TIME`
  - Пришёл позже → `LATE` + "опоздал на X мин/ч"
- Запись создаётся в таблице `attendance`

### 2.4 Уход (Check-Out)
Сотрудник сканирует QR при уходе:
- Система определяет статус ухода:
  - Ушёл раньше окончания графика → `LEFT_EARLY` + "ушёл на X мин/ч раньше"
  - Ушёл вовремя → `ON_TIME`
  - Ушёл позже графика → `OVERTIME` + "задержался на X мин/ч"
- Рассчитывается общее время работы

### 2.5 Правило 23:00
- Если к 23:00 сотрудник не сделал check-out → статус дня → `INCOMPLETE`
- Системный cron в 23:00 обрабатывает незакрытые сессии
- **Исключение (Admin fix):** Сотрудник сообщает о забытом сканировании → Админ в разделе "Сейчас в офисе" указывает время ухода вручную

### 2.6 Ручная правка (Admin)
- Администратор может:
  - Добавить запись об отсутствии с причиной (поле `note`)
  - Исправить время прихода/ухода (`is_manual = true`)
  - Указать время ухода для забытого check-out
- Все ручные правки логируются в `audit_logs`

### 2.7 Офисные сети
- Таблица `office_networks` — список разрешённых IP/подсетей
- Admin может добавлять/удалять/обновлять сети (для нескольких офисов)
- Полное удаление сети из системы (физическое, т.к. это конфигурация)

### 2.8 QR-код офиса
- Один активный QR на офис (или на сеть)
- Только Admin/SuperAdmin может регенерировать QR
- QR хранит зашифрованный токен, привязанный к офисной сети
- При смене офиса/сети — регенерировать QR

### 2.9 Отчёты посещаемости
Фильтры: по неделе | по месяцу | по произвольному периоду | по сотруднику | по группе

Показатели:
- Кто работал больше всего (суммарные часы)
- Кто приходил раньше / уходил позже
- Кто больше пропустил **без причины** (ABSENT / INCOMPLETE)
- Кто больше пропустил **с причиной** (APPROVED_ABSENCE)
- Кто чаще опаздывал (LATE + количество раз)
- Итоговая таблица по каждому сотруднику

**TeamLead** видит отчёты только своей группы.
**Admin / SuperAdmin** видят все.
**INTERN / EMPLOYEE** — не видят отчёты посещаемости других.

## МОДУЛЬ 3 — ДЕЖУРСТВО (DUTY)

### 3.1 Назначение
- **Admin вручную** назначает дежурного на конкретный день
- Один дежурный = один человек на весь день
- Дежурный делает **и уборку, и обед**

### 3.2 Очередь дежурства
- Admin управляет порядком очереди (добавить сотрудника, переставить)
- Система показывает текущую очередь и историю

### 3.3 Обязанности дежурного
**Обед (13:00–15:00):**
- Заказать/купить еду
- Накрыть стол (чай, хлеб, посуда)
- После обеда разделить сумму между всеми и собрать

**Уборка (один день в неделю):**
- Убраться по чеклисту

### 3.4 Чеклист дежурства
- Admin создаёт/редактирует список задач (`duty_checklist_items`)
- Дежурный видит чеклист и отмечает выполненные пункты
- Чеклист сохраняется в `completion_tasks` (JSON)

### 3.5 Подтверждение дежурства
Процесс:
1. Дежурный выполняет задачи, отмечает в чеклисте
2. Дежурный сканирует QR офиса (с офисной сети) → `completion_qr_verified = true`
3. Приходит уведомление Admin: "Сотрудник [ФИО] отметил завершение дежурства — подтвердите"
4. Admin нажимает "Подтвердить" → `verified = true` | или "Отклонить" с комментарием
5. **Если не сканировал, но сделал** — сообщает Admin (вне системы), Admin подтверждает вручную до 12:00

Дедлайн подтверждения: **до 12:00 следующего рабочего дня**

### 3.6 Обмен дежурствами
1. Сотрудник A отправляет запрос на обмен сотруднику B (указывает свою дату и дату B)
2. Сотрудник B принимает или отклоняет (в приложении)
3. Admin получает уведомление об обмене (автоматически записывается)
4. В истории отображается: "X дежурил вместо Y (обмен с Y)"
5. **Одобрение Admin при обмене НЕ требуется** — это автоматически

### 3.7 Если дежурный не сделал дежурство
- К 23:00 дежурство не подтверждено → уведомление Admin
- В отчёте записывается: не сделал дежурство
- Это влияет на рейтинг/историю сотрудника

### 3.8 Главный экран — блок дежурства
На главном экране (Home) отображается:
- "Сегодня дежурит: [ФИО]"
- Если это ты → "Вы сегодня дежурите! Нужно сделать: [список]" + кнопка "Отметить выполнение"

### 3.9 Отчёты дежурств
Фильтры: по сотруднику, по периоду, по статусу (выполнено / не выполнено / передано)

## МОДУЛЬ 4 — НОВОСТИ

### 4.1 Функционал
- Admin создаёт новость: заголовок + текст (+ опционально фото)
- Все сотрудники видят ленту новостей
- Сотрудники могут только читать (без комментариев, без реакций)
- Admin видит: "Прочитали N из M сотрудников" (read tracking)
- Категории: одна общая лента

### 4.2 Read Tracking
- При открытии новости сотрудником → запись в `news_reads` (user_id + news_id + timestamp)
- Admin видит список: кто прочитал, кто нет

### 4.3 Уведомления
- При создании новости → push всем активным сотрудникам

## МОДУЛЬ 5 — УВЕДОМЛЕНИЯ

Push-уведомления (FCM) отправляются в следующих случаях:

| Событие | Получатель |
|---------|-----------|
| Новый сотрудник ждёт подтверждения | Admin |
| Новость опубликована | Все сотрудники |
| Сегодня ваше дежурство | Дежурный |
| Дежурство не подтверждено к 23:00 | Admin |
| Запрос на обмен дежурством | Целевой сотрудник |
| Обмен дежурством принят/отклонён | Инициатор обмена |
| Дежурный отметил выполнение — подтвердите | Admin |
| Заявка на отпуск одобрена/отклонена | Сотрудник |
| Заявка на отпуск подана | Admin |
| Сотрудник забыл check-out (23:00) | Admin (список) |

## БИЗНЕС-ПРАВИЛА (ОБЯЗАТЕЛЬНО соблюдать)

1. **UUID** как primary key везде
2. **Soft delete** через `deleted_at` — физического удаления нет (кроме office_networks)
3. **Пароли** только через bcrypt (cost factor 12)
4. **JWT**: Access 15 мин, Refresh 30 дней
5. **Минимум рабочего дня**: 6 часов — API возвращает ошибку при попытке сохранить расписание < 6 ч
6. **Посещаемость** = QR токен + IP офисной сети (оба фактора обязательны)
7. **Правило 23:00**: незакрытые сессии → статус INCOMPLETE
8. **Все действия Admin** логировать в таблицу `audit_logs`
9. **Мягкое удаление** пользователя — данные сохраняются для истории отчётов
10. **Обмен дежурствами** — без одобрения Admin, автоматически фиксируется
