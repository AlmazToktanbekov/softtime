## 1. Общая информация о проекте

### 1.1 Идентификация проекта

| Параметр | Значение |
|----------|----------|
| Название проекта | SoftTime |
| Компания-заказчик | Softjol |

| Язык интерфейса | Русский (основной), Кыргызский (опционально) |
| Целевой рынок | IT-компании Кыргызстана (первично — Softjol) |

### 1.2 Краткое описание

SoftTime — внутренняя корпоративная платформа: посещаемость и стажёры, дежурства, новости, офисный менеджмент.

- Мобильное приложение (Flutter) — QR, расписание, новости, дежурства.
- Веб-панель администратора + мобильное приложение для администратора — полное управление.

### 1.3 Проблемы и ценность

**Проблемы:** нет прозрачного учёта посещаемости; нет системы дежурств; нет единого канала новостей; нет контроля стажёров; ручное управление офисом.

**Ценность:** автоматический учёт времени, прозрачность, единая точка управления, масштабируемость.

## 2. Цели и задачи проекта

### 2.1 Главная цель

Построить сильную команду и вырасти в аутсорс-компанию через прозрачное управление офисом, стажёрами и сотрудниками.

### 2.2 Бизнес-цели (измеримые)

1. Снизить опоздания и прогулы (~40% за 3 месяца).
2. Каждый стажёр — трекинг прогресса в реальном времени.
3. Автоматизация учёта рабочего времени (ручные записи исключены).
4. База данных кандидатов / история стажёров.
5. Снизить административную нагрузку на тимлидов (3–5 ч/нед).

### 2.3 Технические цели

- REST API + JWT.
- QR + сетевая верификация (офисная сеть).
- Офлайн mobile с синхронизацией (цель).
- Push-уведомления.
- Аналитические отчёты с фильтрами.

### 2.4 KPI

| KPI | Цель | Критично |
|-----|------|----------|
| Регистрация прихода через QR | < 5 с | Да |
| Uptime | ≥ 99,5% | Да |
| Загрузка страниц | < 2 с | Да |
| Покрытие тестами Backend | ≥ 80% | Нет |
| Ошибки QR | < 1% на 1000 сканов | Да |
| Формирование отчёта | < 3 с | Нет |

## 6. Функциональные требования

### 6.1 Аутентификация и регистрация

- Регистрация: ФИО, Email (Gmail), телефон +996…, логин, пароль (мин. 8 символов, 1 заглавная, 1 цифра).
- Уникальность email и логина.
- После регистрации — статус **PENDING**; подтверждение администратором.
- Push админу о новой заявке; при подтверждении — push/email пользователю.
- JWT: Access 15 мин, Refresh 30 дней; автообновление; logout с инвалидацией refresh.
- 5 неудачных попыток входа → блокировка 15 минут.

**Статусы аккаунта:** PENDING, ACTIVE, LEAVE, WARNING, BLOCKED, DELETED.  
**Важно:** soft delete (`deleted_at`); физическое удаление через 90 дней (крон / SUPER_ADMIN).

### 6.2 Посещаемость

**Два обязательных фактора:** валидный QR + IP в офисной сети. Иначе отметка не засчитывается; при сбое Wi‑Fi — ручная запись админом.

**QR:** офис/помещение, токен, ACTIVE/INACTIVE; регенерация, печать.

**Офисные сети:** IP-диапазоны по офисам; проверка по IP (SSID в API недоступен).

**График:** по дням недели; минимум **6 часов** в рабочий день; выходные — отметка не обязательна.

**Приход:** ON_TIME, LATE, EARLY (в ТЗ — ранний приход).  
**Уход:** LEFT_EARLY, ON_TIME, OVERTIME.  
**Правило 23:00:** нет ухода до 23:00 → **INCOMPLETE** (крон ~23:05).

**Ручная корректировка** админом с аудитом.

**Статусы посещаемости (агрегированно в ТЗ):** PRESENT, LATE, LEFT_EARLY, OVERTIME, EARLY_ARRIVAL, INCOMPLETE, ABSENT_EXCUSED, ABSENT_UNEXCUSED, ON_LEAVE — см. согласование с enum в БД реализации.

### 6.3 Дежурство

Очередь, чеклист задач, QR завершения, подтверждение админом/тимлидом, обмен дежурствами, обеденное дежурство (v1.1) — как в разделах 6.3 ТЗ.

### 6.4 Новости

Посты, типы, расписание публикации, read tracking; комментарии не в MVP.

### 6.5 Профиль сотрудника

Поля: фото, ФИО, email, телефон, логин, роль, команда, дата начала, тип EMPLOYEE/INTERN, статус, график, тимлид для стажёров, комментарий админа — см. таблицу 6.5 ТЗ.
